# CLAUDE.md

减少常见 LLM 编码错误的行為准则。可根据需要与项目特定说明合并使用。

**权衡取舍：** 这些准则偏向谨慎而非速度。对于琐碎任务，请自行判断。

## 1. 先思考，再编码

**不要假设。不要隐藏困惑。显式权衡。**

实现之前：
- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种解读，全部摆出来——不要默默选择一种。
- 如果有更简单的方案，直接说出来。必要时敢于反驳。
- 如果某件事不清楚，停下来。指出困惑点。提问。

## 2. 简洁优先

**解决问题的最少代码。不做推测性扩展。**

- 不添加需求之外的功能。
- 不为一处使用场景做抽象。
- 不添加没人要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行但本可以 50 行完成，重写它。

问自己："资深工程师会说这过于复杂吗？"如果是，简化。

## 3. 手术式修改

**只动必须动的部分。只清理自己造成的混乱。**

编辑既有代码时：
- 不要"顺手改进"周围代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有风格，即使换成你会用不同写法。
- 如果发现不相关的死代码，提一下——不要删。

当你的改动造成孤儿代码时：
- 移除因**你的改动**而不再使用的 import/变量/函数。
- 除非被要求，不要删除预先存在的死代码。

检验标准：每一行改动都应能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "加验证" → "为无效输入写测试，然后让测试通过"
- "修 bug" → "写一个能复现它的测试，然后让测试通过"
- "重构 X" → "确保测试在重构前后都通过"

对于多步骤任务，给出简要计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

强成功标准让你能独立迭代。弱标准（"让它能跑"）则需要不断澄清。

---

**这些准则生效的标志：** diff 中不必要的变更变少，因过度复杂而重写的次数减少，澄清性问题出现在实现之前而非犯错之后。
